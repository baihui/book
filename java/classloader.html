ClassLoader

类加载器是加载一个类到虚拟机中的，但它在虚拟机中却远远不限于类加载阶段如确定一个类在虚拟机中的唯一性必须与之关联，且每一个类加载器都有独立类名称空间

  • jvm中存在4类加载器

    • 启动类加载器 主要加载系统核心类库如String,Object，并且只能由JVM使用,如下获取类加载器，是无法获取到启动类加载器 

      
  B b = new B() ;
  System.out.println(b.getClass().getClassLoader());
  System.out.println(String.class.getClassLoader());
  System.out.println(Map.class.getClassLoader());

    • 控制类加载器 是一个可以指定加载路径的类加载器 
        -Xbootclasspath/a/b

    • 应用类加载器

    • 自定义类加载 new ClassLoader()实现loadClass方法

不是同一个类加载器会出现如下情况：


ClassLoader classLoader = new ClassLoader() {
    @Override
    public Class<?> loadClass(String name) throws ClassNotFoundException {

        InputStream inputStream= null;
        try {

            inputStream = new FileInputStream(new File(name));
            byte[] b = new byte[inputStream.available()];
            inputStream.read(b);
            return (Class<?>) defineClass("org.bh.controller.B", b, 0, b.length) ;

        } catch (Exception e) {
            if (inputStream == null){
                return super.loadClass(name) ;
            }
        }
        return null ;
    }
} ;

Object a = classLoader.loadClass("~/gitHub/eventController/src/org/bh/controller/B.class").newInstance() ;

//B 由系统默认的
System.out.println("a由自定义类加载器加载"+ (a instanceof B));

B b = new B() ;
System.out.println("默认的"+(b instanceof B));

自定义类加载器false
默认的true

确定一个类是否相等首要条件是类加载器为同一个

  • 双亲委托模式

    在类加载的时候默认会使用双亲委托模式，判断当前类是否已经在方法区中，如果已经存在返回可用的类，否者委托父类加载器也会委托父加载器直至到启动类加载器进行加载，如果加载失败会从上到下进行加载直到自己加载如果还失败异常，这就是双亲委托模式，

这也就说明为什么如下会出现这种情况了


//B 由系统默认的
System.out.println("a由自定义类加载器加载"+ (a instanceof B));


打印：
自定义类加载器false
默认的true

首先在实现自定义类加载器中并没有委托父加载器去加载，其次是因为虚拟机中确定一个类的唯一性前提条件是在相同的类加载器，而（a instanceof B）使用了不同的类加载器，所以不相等也就是无法强制转换。

  • 热替换

可以自定义类加载器实现热替换即在每次调用之前重新创建类加载器以及替换的类，但是注意一点类逃逸出现带来的内存空间的浪费
