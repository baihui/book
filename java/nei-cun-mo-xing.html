内存模型

内存模型

内存模型规定了所有的变量都存储在主内存中,每条线程还有自己的工作内存，线程的工作内存中保存了被该线程使用到的变量的主内存副本拷贝 ，线程对变量的所有操作（读取、赋值等）都必须操作变量在多线程下必须在工作内存中进行，而不能直接读写主内存中的变量 不同线程之间也无法直接访问对方工作内存中的变量，线程间变量值的传递均需要通过主内存来完成交互的，为了提高性能，编译器和处理器常常会对指令做重排序,分3种类型：

1）编译器优化的重排序。编译器在不改变单线程程序语义的前提下，可以重新安排语句的执行顺序。
2）指令级并行的重排序。现代处理器采用了指令级并行技术来将多条指令重叠执行。如果不存在数据依赖性，处理器可以改变语句对应机器指令的执行顺序。
3）内存系统的重排序。由于处理器使用缓存和读/写缓冲区，这使得加载和存储操作看上去可能是在乱序执行。

会导致线程之间，对共享变量可见性问题如下：

  • 缓冲好处带来的问题

工作内存实际上就是处理器的缓冲，每个处理器通过写缓冲来减少对主内存操作的等待，最后通过批量（多个指令合并，在一个总线事务读/写入）刷回主内存 。这个特性会对内存操作的执行顺序产生重要的影响,处理器对内存的读/写操作的执行顺序，不一定与内存实际发生的读/写操作顺序一致！


    int a = 0 ,b = 0 ;

    //线程A
    public void setAB(){
        a = 1 ;
        b = a ;
    }

    //线程B
    public void setBA(){
        b = 1 ;
        a = b ;
    }

同时运行输出多种情况

  • 在第一次打印a和b输出1
  • 在第二次打印a和b输出可能都会是0，也有可能是0和1,出现这种情况是由于缓冲造成的 线程A和B在写入主内存顺序不确定，导致线程之间出现了变量可见性问题 如图：

在图上是解释了缓冲影响了多线程之间数据可见性，但是在一个线程中不会存在以上问题

  • 重排序带来的问题
    为了提高性能编译器和处理器会对指令重排序，但是会遵守数据依赖性不会改变存在数据依赖关系的两个操作的执行顺序，数据依赖指的是两个操作（指令）对一个变量进行一个读一个写，那么他们之间存在依赖关系。

    • 两个操作a和b存在如下关系不进行排序

      | 变量 | 依赖关系 | 重排序 | | :---: | :---: | :---: | | a=1;b=a; | b依赖于a | 不排序 | | a=1;a=2; | 对a的后一个写依赖前一个写 | 不排序 | | a=b;b=1; | 读一个变量b之后，在写b | 不排序 | | a=1;b=1; | 不存在依赖 | 重排序 |

    int aIf = 0 ;
    boolean boo = false ;
    public void setaIf(){
        aIf = 1 ; //a
        boo = true ; //b
    }

    public void aVoid(){

        if (boo){
            int c = aIf *2 ;
            System.out.println(c) ;
        }
    }


// a和b操作之间不存在依赖关系导致重排序，会打印0
  • 内存模型规定对共享变量的操作需通过主内存完成，操作是原子性的，内存可见性问题通过禁止编译器重新排序和插入内存屏障指令来实现

原子性定义了变量主内存到工作内存的操作是原子的（long和duoble除外）指令

  • lock（锁定）：作用于主内存的变量，它把一个变量标识为一条线程独占的状态
  • unlock（解锁）：作用于主内存的变量，它把一个处于锁定状态的变量释放出来，释放后的变量才可以被其他线程锁定
  • read（读取）：作用于主内存的变量，它把一个变量的值从主内存传输到线程线程的工作内存中，以便随后的load动作使用。
  • load（载入）：作用于工作内存的变量，它把read操作从主内存中得到的变量值放入工作内存的变量副本中。
  • use（使用）：作用于工作内存的变量，它把 工作内存中一个变量的值传递给执行引擎，每当虚拟机遇到一个需要使用到变量的值的字节码指令时将会执行这个操作。
  • assign（赋值）：作用于工作内存的变量，它把一个从执行引擎接收到的值赋给> 工作内存的变量，每当虚拟机遇到一个给变量赋值的字节码指令时执行这个操作。
  • store（存储）：作用于工作内存的变量，它把工作内存中一个变量的值传送到主内存中，以便随后的write操作使用
  • write（写入）：作用于主内存的变量，它把store操作从工作内存中得到的

  • 禁止编译器重排序和插入内存屏障指令来实现顺序性和内存可见性，内存屏障指令如下几种：

    • loadload

      a = c ; loadload ; b = c; ... ; 确保a的装载比后续任何装载优先 。

    • storestore

      a = c ; storestore ; b = c; ... ; 确保a的刷新主内存比后续任何存储优先

    • loadstore

      a = c ; loadstore ; b = 10; ... ; 确保a的装载比后续任何存储优先

    • storeload

      a = 10 ; storeload ; b = c; ... ; 确保a的刷回比后续任何装载优先

内存屏蔽指令可以解决禁止重排序以及对缓冲的造成写主内存的顺序不一致问题

  • volatile 是这样定义的当写一个volatile变量时，java内存模型会把该线程对应的本地内存中的共享变量值刷新到主内存，当读一个volatile变量时会将它的本地设置无效，直接读主内存，实现原理是通过插入内存屏障指令完成的具体如下：

    • 每个volatile写操作的前面插入一个StoreStore屏障。
    • 每个volatile写操作的后面插入一个StoreLoad屏障。
    • 每个volatile读操作的后面插入一个LoadLoad屏障。
    • 每个volatile读操作的后面插入一个LoadStore屏障。

综上所述：内存模型通过happens-before(先行发生)的概念来阐述操作之间的内存可见性。如果一个操作执行的结果需要对另一个操作可见，那么这两个操作之间必须要存在happens-before关系具体如下：

  • 程序顺序规则
    一个线程中的每个操作，可见于该线程中的任意后续操作。

  • 锁的规则
    synchronized对一个锁的解锁，可见于随后对这个锁的加锁。当线程获取锁时，JMM会把该线程对应的本地内存置为无效，使得被监视器保护的临界区代码必须从主内存中读取共享变量

  • volatile

    • 对一个volatile域写，可见于随后对这个域得读。

    • 当写一个volatile变量时，JMM会把该线程对应的本地内存中的共享 变量值刷新到主内存（堆内存中）。复合操作不是，例如 volatile x ++ 修饰的对象变量 只会对变量指向的引用有效

  • 传递性

    如果A可见于B 且B可见于C 那么A对C是可见的。

两个操作具有可见关系。并不意味着请一个操作必须在后一个操作之前执行。可见关系定义了，一个操作的结果对后一个操作是可见的。
