commit

git是一个内容寻址文件系统,也就是一个key-value的健-值对数据库可以向数据库插入任意类型的内容，它会返回一个key 通过给key可以在任意时刻查询该key对应的内容,通过不同的对象来封装内容和目录，git中存在两种类型对象：

  • 数据对象(blob Object) --添加暂存区操作会创建数据对象

    数据对象用于记录文件的内容（value）存放于git数据库中，,内容信息加上特定的头部信息一起hahs生成校验和(key)，这key包含40个字符其中前两个字符为子目录名称后面38个为文件名

添加暂存区操作会为每个文件创建key也就是校验并且将这个key添加到暂存区中，然后将当前文件的快照保存在.git/objects 中


echo '添加到git中的内容' | git hash-object -w --stdin


f9ac68a46a81bcd37efdcf9c1061e09e1b8d8a38

f9ac68a46a81bcd37efdcf9c1061e09e1b8d8a38 是生成的校验和其中f9是子目录ac68a46a81bcd37efdcf9c1061e09e1b8d8a38i 是文件名称

文件存放路径 如下：


find .git/objects -type f
.git/objects/f9/ac68a46a81bcd37efdcf9c1061e09e1b8d8a38

通过key获取内容

git cat-file -p f9ac68a46a81bcd37efdcf9c1061e09e1b8d8a38i

添加到git中的内容


echo '123' > a.md

git hash-object -w a.md

9ed5af45c54ef0005388e4d564361f48df5c6d01

git cat-file -p 9ed5af45c54ef0005388e4d564361f48df5c6d01

rm -rf a.md

  • 树对象(tree Object) -- 在进行提交时创建树对象

    树对象是通过暂存区在某一时刻所表示的状态来创建的，用于保存目录以组织多个文件的记录对象。 当使用 git commit 进行提交操作时，Git 会先计算每一个子目录的校验和，然后在 Git 仓库中这些校验和保存为树对象。

例如: 创建三个文件添加到暂存区中,并且生成树对象


  echo 'i is A' > a
  echo 'i is B' > b
  echo 'i is C' > c

  -- 添加到暂存区中
  git update-index --add a b c

  -- 创建树对象返回该树对象对应的key
  git write-tree

  b31326ecd6e8d776c775b16138b207296cd67f03

  --通过树对象查询包含的数据对象
  git cat-file -p b31326ecd6e8d776c775b16138b207296cd67f03i

  • 提交对象(commit tree) -- 创建完树对象后在创建提交对象

    关联了树对象指针，并且记录了谁保存，为什么保存以及保存时间的对象

如：创建一个提交对象


-- 新建文件，添加到暂存区中

Git所做的实质工作—将被改写的文件保存为数据对象，更新暂存区，记录树对象，最后创建一个指明了顶层树对象和父提交的提交对象。

  • 引用对象

如果想浏览整个提交记录 ，我们需要记住最后一次提交对象的key,如访问最后提交的历史记录:


git log --stat 8d71357de4b799cf4cb3505afa022e04c5767f46

通过定义一个文件名称来代替最后提交的历史的key值，文件存放refs 目录中


echo '8d71357de4b799cf4cb3505afa022e04c5767f46' >.git/refs/heads/master

git log

在heads目下创建来一个maste文件，文件内容包含了最后一次提交的对象的key

等价与下面命令(官方推荐使用下面方式来关系引用对象)


git update-ref refs/heads/master 8d71357de4b799cf4cb3505afa022e04c5767f46

当我们创建分支实际是在heads目录下创建一个分支名称的文件，文件内容是通过HEAD指向的当前分支文件获取的提交对象

如：新创建分支(git branch tex),实际指向了 如下底层命令


-- 获取HEAD 指向的引用对象
cat .git/HEAD
ref: refs/heads/master


-- 获取最后一次提交对象key
cat refs/heads/master
8d71357de4b799cf4cb3505afa022e04c5767f46

-- 在heads下创建名称为tex文件并且内容是'8d71357de4b799cf4cb3505afa022e04c5767f46'
git update-ref refs/heads/tex  8d71357de4b799cf4cb3505afa022e04c5767f46
  • HEAD引用是一个符号引用包含一个引用对象,通过checkout切换分支来更新指向的引用对象,也就是当前所在的分支文件

如：当前分支是tex

cat .git/HEAD

➜  Downloads git:(tex) ✗ cat .git/HEAD
ref: refs/heads/tex

git 是一个key-value 的数据库 ，通过添加内容到数据库返回key 来完成，一次完成的提交包含如下步骤: （1） update-index 添加到暂存区中，生产对对象，数据对象 ，这个过程包含如下

  1. hash-object 添加数据库 生成数据对象
  2. write-tree 创建树对象 记录所有数据对象的key

（2）commit-tree 创建提交对象，更新引用对象内容

  1. 创建提交对象(记录为什么提交 ，谁提交以及日期) 和包含上一次提交对象的key（通过HEAD获取）
  2. 更新引用对象 refs/heads/分支文件中包含的最后一次提交对象记录
  • 标签对象

标签对象（tag object）非常类似于一个提交对象—它包含一个标签创建者信息、一个日期、一段注释信息，以及一个指针。 主要的区别在于，标签对象通常指向一个提交对象，而不是一个树对象。

它像是一个永不移动的分支引用—永远指向同一个提交对象，只不过给这个提交对象加上一个更友好的名字罢了。

  • 远程引用 远程引用会记录最近一次推送时每一个分支所对应的值也就是最后一次提交对象的值，并保存在refs/remotes/远程仓库名称/分支名称

如 cat .git/refs/heads/qxb-20181225-stage3最后一次提交对象的值是 ： c60d3be7a3f1b38abb589508a350453e9af373c4

执行推送操作后远程引用对象中记录也是该值


git push origin

cat .git/refs/remotes/origin/qxb-20181225-stage3
c60d3be7a3f1b38abb589508a350453e9af373c4

  • 一个文件一次提交最少会.git/objects/目录下创建三个对象：数据对象，树对象和提交对象

    • 数据对象：一个文件对应一个对象
    • 树对象：多个数据对象对应一个树对象
    • 提交对象：对应一个树对象

  • 另外对象如：符号引用对象和引用对象或者

    • 符号引用对象会在：.git/HEAD 目录下创建一个指向引用对象的文件
    • 引用对象：在.git/refs/heads/分支文件 中记录最后一次的提交对象

  • 打包

Git 最初向磁盘中存储对象时所使用的格式被称为“松散（loose）”对象格式。 1 Git 会时不时地将多个这些对象打包成一个称为“包文件（packfile）”的二进制 文件，以节省空间和提高效率。 2 当版本库中有太多的松散对象，或者你手动执行 git gc 命令，或者你向远程服务器执 行推送时，Git 都会这样做。 要看到打包过程，你可以手动执行 git gc 命令让 Git 对对象进行打包：


➜  git git:(master) git gc
Counting objects: 56, done.
Delta compression using up to 4 threads.
Compressing objects: 100% (31/31), done.
Writing objects: 100% (56/56), done.
Total 56 (delta 24), reused 50 (delta 21)
